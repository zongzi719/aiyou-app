 # Luna 发布打包命令速查

> 位置：项目根目录  
> 目的：一眼看懂「升级版本」与「打包发布」命令，避免版本号混乱。

---

## 🟦 一、先理解三类命令

- 🟦 **仅升级版本**：只改版本号，不打包  
- 🟩 **升级版本 + 打包**：自动升版本，然后打包  
- 🟨 **仅打包**：不改版本号，直接重打当前版本

### 名词解释（先看这个）

- **小升级（patch）**：最后一位 `+1`，例如 `1.2.0 -> 1.2.1`  
  适用：修 bug、小优化、文案/样式微调。
- **中升级（minor）**：中间位 `+1`，最后一位清零，例如 `1.2.1 -> 1.3.0`  
  适用：新增功能、较明显功能变化。

---

## 🟦 二、仅升级版本（不打包）

### 1) 小升级（patch）

```bash
npm run version:bump:patch
```

- 规则：`1.2.0 -> 1.2.1`
- 同时：
  - iOS `CURRENT_PROJECT_VERSION` 自动 `+1`
  - iOS `buildNumber` 自动 `+1`
  - Android `versionCode` 自动 `+1`
- 适用场景：今天主要是修复问题、微调功能。

### 2) 中升级（minor）

```bash
npm run version:bump:minor
```

- 规则：`1.2.1 -> 1.3.0`
- 同时：
  - iOS `CURRENT_PROJECT_VERSION` 自动 `+1`
  - iOS `buildNumber` 自动 `+1`
  - Android `versionCode` 自动 `+1`
- 适用场景：今天有新功能上线，想与上个版本明显区分。

---

## 🟩 三、升级版本 + 打包（推荐发新包）

### 1) iOS TestFlight（小升级 patch）

```bash
npm run release:ios:testflight:patch
```

- 先升 patch，再打 iOS production，并自动提交 TestFlight
- 适用场景：最常用，日常提测。

### 2) iOS TestFlight（中升级 minor）

```bash
npm run release:ios:testflight:minor
```

- 先升 minor，再打 iOS production，并自动提交 TestFlight
- 适用场景：本次有新增功能，需要版本号更明显。

### 3) Android 生产包（小升级 patch）

```bash
npm run release:android:prod:patch
```

- 先升 patch，再打 Android production
- 适用场景：Android 日常发包/提测。

### 4) Android 生产包（中升级 minor）

```bash
npm run release:android:prod:minor
```

- 先升 minor，再打 Android production
- 适用场景：Android 新功能版本发布。

---

## 🟨 四、仅打包（不升级版本）

### 1) iOS TestFlight

```bash
npm run build:ios:testflight
```

- 适用于：同一版本重打包、重传

### 2) Android 生产包

```bash
npm run build:android:prod
```

- 适用于：同一版本重打包

---

## 🟥 五、常见错误与处理

### 错误：`You've already submitted this build of the app`

- 原因：提交了重复构建号（iOS `CFBundleVersion` 未变化）
- 处理：执行一次版本升级命令后再打包（至少让 build 号 +1）

推荐：

```bash
npm run release:ios:testflight:patch
```

## 🟫 六、开发调试常用命令

### 真机（Dev Client，LAN 模式，推荐，**iOS 与 Android 通用**）

```bash
cd /Users/ZHOU/Desktop/project/luna-main
npx expo start --dev-client --host lan -c
```

- **说明**：同一套 Metro 供两侧 Dev Client 连接，不区分 iOS / Android

### 真机（原脚本，tunnel 模式，**iOS 与 Android 通用**）

```bash
npm run start:dev-client -c
```

- **说明**：同上，Metro 对 iOS / Android Dev Client 通用
- 适用：不在同一局域网时临时使用
- 注意：依赖 ngrok，偶发连不上属于常见情况

## 🟪 七、建议使用顺序

### 发新 iOS 测试包（最常用）

```bash
npm run release:ios:testflight:patch
```

### 发新 Android 生产包

```bash
npm run release:android:prod:patch
```

### 不改版本，仅重打当前包

```bash
npm run build:ios:testflight
# 或
npm run build:android:prod
```

### Android 产物说明

`eas.json` 中 `production` 的 Android 为 **`buildType: apk`**，`npm run build:android:prod` / `release:android:prod:*` 产物为 **.apk**（未配置 AAB）。

## 八、EAS 云打包与本地 .env（必读）

`EXPO_PUBLIC_*` 在 **EAS 云端** 打包时由 Metro **写入 JS Bundle**；云端 **不会** 读取你本机被 gitignore 的 `.env`（与 iOS / Android 无关，两边共用同一套打包注入逻辑）。

- **已做仓库配置**：`eas.json` 中各 `build` profile 已设 `environment`（`development` / `preview` / `production`），与 [Expo 项目 → Environment variables](https://expo.dev) 中同名环境一一对应。
- **你遇到的现象**（本机正常、TestFlight/商店包缺 Key）：说明 **EAS 上 `production` 环境** 未包含与本地 `.env` 相同的 `EXPO_PUBLIC_Aliyun / EXPO_PUBLIC_TENCENT_…` 等变量。打包日志里若只出现百炼等少数变量，其余都缺，即为未在 EAS 中创建。
- **解决步骤**（与官方文档一致，[Environment variables in EAS](https://docs.expo.dev/eas/environment-variables/)）  
  1. 在 **本机** 已有一份完整根目录 `.env`（含所有 `EXPO_PUBLIC_*`）。  
  2. 将其中变量同步到 EAS 的 **production**（三选一或组合）：在 Expo 网站 **Add variables** 批量粘贴，或跑下面的脚本，或逐条 `eas env:create --environment production`。  
  3. **必须重新执行一次** `npm run build:ios:testflight` / `build:android:prod`，旧包不会自动更新内联的 Key。

**用脚本从本机 .env 推到 EAS（推荐一次对齐）**

```bash
# 先看将要执行的命令
npm run eas:env:sync-prod:dry-run

# 已 eas login 后，真正写入 EAS production
npm run eas:env:sync-prod
```

**核对 EAS 上 production 与本地是否一致**（可拉取为 `.env.local` 对比，勿提交该文件）：

```bash
npm run eas:env:pull:production
```

**本次主流程会用到、且你截图里已报缺的项（务必在 EAS production 中齐全）**  

例如：`EXPO_PUBLIC_TENCENT_MAAS_API_KEY`（图像/TokenHub）、`EXPO_PUBLIC_ALIYUN_OSS_ACCESS_KEY_ID` / `EXPO_PUBLIC_ALIYUN_OSS_ACCESS_KEY_SECRET` / `EXPO_PUBLIC_ALIYUN_OSS_BUCKET`、以及你声音复刻路线所需的 DashScope / 腾讯云 等。具体以你根目录 `.env` 为准，与 [lib/envHealth.ts](lib/envHealth.ts) 的校验项对照即可。

> 说明：依赖 `app.config.js` 的变量在 **预构建/配置解析** 阶段即需要存在于构建环境，同样要出现在 EAS 的对应 environment 中。代码里请使用 **`process.env.EXPO_PUBLIC_xxx` 的完整点号写法**，勿对 `process.env` 用可选链，否则可能无法被 Metro 正确替换（见 [Ex 官方指南](https://docs.expo.dev/guides/environment-variables/)）。
